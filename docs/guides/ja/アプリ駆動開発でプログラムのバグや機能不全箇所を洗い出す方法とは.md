## 📋 1. テスト観点の全体マップ

### **機能テスト（正しく動くか）**

- ✅ データ処理の正確性

### **非機能テスト（どれだけ良く動くか）**

- ⚡ パフォーマンス（速度・応答時間）
- 🔒 セキュリティ（認証・認可・暗号化）
- 📈 スケーラビリティ（負荷耐性）
- 🔄 可用性・信頼性（稼働率・復旧）
- 🌐 互換性（環境・ブラウザ・デバイス）

## 🎯 2. 開発段階別の洗い出し手法

### **【設計・実装段階】**

**静的解析**

- コードレビュー（ペアプログラミング、プルリクエスト）
- 静的解析ツール（SonarQube、ESLint、Pylintなど）
- 設計書とコードの乖離チェック
- アーキテクチャ違反の検出

**ユニットテスト**

- 関数・メソッド単位のテスト
- テストカバレッジ測定（80%以上が目安）
- モック・スタブを使った独立テスト
- TDD（テスト駆動開発）の採用

### **【結合段階】**

**統合テスト**

- APIの入出力検証
- データベース連携テスト
- 外部サービス連携テスト
- マイクロサービス間通信テスト

**回帰テスト**

- 既存機能への影響確認
- 自動テストスイートの実行
- CI/CDパイプラインでの継続的検証

### **【リリース前段階】**

**システムテスト**

- エンドツーエンドテスト（E2E）
- シナリオベーステスト
- 探索的テスト（自由な操作での発見）
- UAT（ユーザー受け入れテスト）

**非機能テスト**

- 負荷テスト（想定ユーザー数での動作）
- ストレステスト（限界値の確認）
- セキュリティテスト（脆弱性診断）
- パフォーマンステスト

### **【リリース後】**

**本番監視**

- リアルタイムモニタリング
- エラーログ分析
- ユーザー行動分析
- クラッシュレポート収集

## 🔍 3. バグの種類別検出アプローチ

### **機能バグ**

```
検出方法：
・テストケース実行（正常系・異常系）
・境界値テスト
・同値分割法
・デシジョンテーブル
```

*例：ECサイトで割引計算が間違っている*

### **UI/UXバグ**

```
検出方法：
・視覚的回帰テスト（スクリーンショット比較）
・レスポンシブデザインチェック
・ユーザビリティテスト
・アクセシビリティ監査（WCAG準拠）
```

*例：モバイルでボタンが隠れる、色のコントラスト不足*

### **データバグ**

```
検出方法：
・データバリデーションテスト
・データ整合性チェック
・バックアップ/リストアテスト
・マイグレーションテスト
```

*例：顧客データが重複する、金額計算の丸め誤差*

### **パフォーマンスバグ**

```
検出方法：
・プロファイリングツール
・ボトルネック分析
・メモリリーク検出
・ネットワーク遅延測定
```

*例：画像読み込みが遅い、メモリ使用量増大*

### **セキュリティバグ**

```
検出方法：
・脆弱性スキャン（OWASP Top 10）
・ペネトレーションテスト
・認証・認可テスト
・暗号化検証
```

*例：SQLインジェクション、XSS、情報漏洩*

### **統合バグ**

```
検出方法：
・APIコントラクトテスト
・サービス間通信テスト
・タイムアウト・リトライテスト
・バージョン互換性テスト
```

*例：API仕様変更で連携エラー、タイムアウト発生*
