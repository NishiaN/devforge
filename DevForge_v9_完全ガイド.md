## ①基礎概念 — DevForgeとは何か

### 1-1. 一言で言うと

**「質問に答えるだけで、AIが読める開発設計書を225+ファイル自動生成するツール」**

コードは一行も書かない。ブラウザで動作する単一HTMLファイル。インストール不要。

---

### 1-2. なぜ必要か — 2026年の「コンテキスト断片化」問題

AI開発ツール（Cursor / Claude Code / Windsurf / GitHub Copilot）が普及した結果、皮肉にも新しい問題が生まれた。

```
月曜: CursorでNext.jsの認証を実装 → コードが生成された
水曜: Claudeにデータベース設計を依頼 → 別のスキーマが出てきた
金曜: Windsurf でAPIエンドポイントを追加 → 3つのスキーマが矛盾
```

**どのAIも「あなたのプロジェクト全体」を知らない。** 毎回「私のアプリはこういう設計で、技術スタックはこれで、認証はこうで……」と説明し直す必要がある。

DevForgeはこの問題を解決する：

```
DevForgeで一度設計を整理
    ↓
225+ファイルの「AIが読む設計書」を自動生成
    ↓
すべてのAIツールが「このプロジェクトの全体像」を即座に理解
    ↓
矛盾のない一貫した開発が継続できる
```

### 1-4. 使用者目線 vs 開発者目線

| 視点 | 主な用途 | 必要な知識 |
|------|---------|-----------|
| **使用者（アプリ開発者）** | 自分のアプリ設計書を生成してAIに渡す | プログラミング知識は不要。質問に答えるだけ |
| **開発者（DevForge自体を改造）** | 新ピラー追加・新プリセット作成・ドメイン拡張 | JavaScript（Vanilla）、Node.js、テスト（node:test）|

本ガイドは**両方の視点**を扱う。まず「使用者」として使いこなし、必要なら「開発者」として拡張する。

### 1-5. なぜDevForgeは唯一無二か — 競合不在の10の理由

| # | 特徴 | bolt.new / Lovable | Cursor / Copilot | create-next-app | DevForge v9 |
|---|------|-------------------|-----------------|-----------------|-------------|
| 1 | AIツール向けルール同時生成 | 0 | 0 | 0 | **10カテゴリ（13+ファイル）** |
| 2 | MCP設定自動生成 | × | × | × | **○** |
| 3 | 32ドメインコンプライアンス | × | × | × | **○** |
| 4 | オフライン単一HTML | × | × | × | **○（〜5716KB）** |
| 5 | プリセット数 | ~5 | 0 | ~3 | **859種（標準257+専門602）** |
| 6 | 技術互換性ルール（Why?付き） | 0 | 0 | 0 | **320ルール（33E+143W+144I）** |
| 7 | スキルレベルシステム | × | × | × | **7段階** |
| 8 | UI/生成言語独立バイリンガル | × | × | × | **○** |
| 9 | 設計書一括生成 | 0ファイル | 0ファイル | ~3ファイル | **225+ファイル** |
| 10 | 自動アーキテクチャ整合性監査 | × | × | × | **○（10点満点スコア）** |

#### 各ポイントの解説

1. **10カテゴリAIルール同時生成**: Cursor / Claude Code / Windsurf / Cline / Copilot / Kiro / Codex / Gemini / AI Hooks / Skills — 全ツール向けルールをプロジェクト固有に生成。他ツールでは0。

2. **MCP設定自動生成**: Model Context Protocol のサーバー設定を技術スタックに合わせて自動生成。競合ツールでは対応なし。

3. **32ドメインコンプライアンス**: HIPAA / PCI DSS / ISMAP / APPI / OWASP等、業種別規制フレームワークを自動マッピング。fintech・health・gov等の業種でコンプライアンス要件が自動注入される。

4. **オフライン単一HTML**: APIキー・サーバー・課金不要。エアギャップ環境でも動作。〜5716KBの単一ファイルがすべての機能を内包（CI上限6500KB）。

5. **859プリセット**: 標準257＋専門分野602、44カテゴリ×各2〜6種。ワンクリックで26問の回答が完了。

6. **320互換性ルール（33E+143W+144I）**: 全ルールに「▶ なぜ？」展開カード付き。ERRORは生成ブロック（ガードレール設計）、WARNは承知して続行可能。

7. **7段階スキルシステム**: Lv.0（完全初心者）〜Lv.6（伝道者）でUI・選択肢・生成内容が全変動。初心者にはFirebaseを強制、上級者にはKubernetesやGoを解放。

8. **バイリンガル独立設定**: UIは日本語、生成は英語（またはその逆）が可能。海外チームへの展開・英語圏向けドキュメント生成も対応。

9. **225+ファイル一括生成**: 28柱の設計書・設定・ドキュメントを10〜30秒で生成。bolt.newが「コードを書く」なら、DevForgeは「AIがコードを書くための設計図」を生成する。

10. **アーキテクチャ整合性監査**: 生成後に自動監査→10点満点スコア（ORM/Auth/CORS/async/soft-delete）。`docs/82_architecture_integrity_check.md`に常時生成。

#### 「Phase 0」ポジショニング

> DevForgeはコードを生成しない。代わりに **「AIがコードを書くための設計図」** を生成する。
>
> bolt.new / Lovable が「Phase 1: コード生成」なら、DevForge は「**Phase 0: 設計・仕様・設定**」。
> 競合ではなく補完関係。**DevForge → bolt.new/Cursor/Claude Code の順で使うと最大効果**。
>
> ```
> DevForge（Phase 0: 設計）→ Cursor/Claude Code（Phase 1: 実装）→ デプロイ
> ```

### 2-2. スキルレベルの設定

DevForgeには**7段階のスキルレベルシステム**がある。最初に自分のレベルを選ぶことで、質問の難易度・選択肢・生成されるドキュメントの詳細度が自動調整される。

| レベル | 称号 | 説明 | 主な違い |
|-------|------|------|---------|
| Lv.0 | 🔰 完全初心者 | HTML/CSSを触り始めた | saasプリセット強制、Firebase自動設定、最小限の質問 |
| Lv.1 | 🌱 入門者 | プログラミング基礎は理解 | beginner向け選択肢、6個のKPIチップ |
| Lv.2 | 📗 始めたばかり | 小さなアプリを作ったことがある | 8個のKPIチップ、少し広い選択肢、ERRORゲート有効 |
| Lv.3 | ⚡ 中級者 | React/Node.jsで開発経験あり | 全intermediate選択肢、10個のKPIチップ |
| Lv.4 | 🔥 上級者 | フルスタック開発経験あり | 確認ダイアログスキップ、上級選択肢 |
| Lv.5 | 💎 エキスパート | 本番環境経験・チームリード | pro全選択肢（Go/Python/Kubernetes等）|
| Lv.6 | 👑 伝道者 | 技術コミュニティへの貢献 | エバンジェリストセクション生成 |

**スキルレベル設定の場所**: トップ画面の3つのカード（初心者/中級者/プロ）の下にある**スライダー**でLv.0〜6を細かく設定できる。

> **Tip**: 迷ったら Lv.1（初心者カードを選択）から始める。いつでも変更可能。Lv.2以上になると、互換性チェックのERRORゲート（後述）が有効化され、問題のある技術スタックを選んだ場合に生成が一時ブロックされる。

### 2-4. プリセットの選択肢を把握する（標準257種 + 専門分野602種）

白紙から始める前に、あなたのプロジェクトに近いプリセットを選ぶと設定が大幅に時短できる。

プリセット選択後は**アンドゥ機能**が利用可能（トーストに「元に戻す」ボタンが表示）。プリセット選択前の回答状態をスナップショットとして保持しているため、誤って上書きした場合でも即座に復元できる。

> **✨ 新機能: プリセット推薦ウィザード**: 「どれを選べばいいかわからない」という場合は、プリセット画面の「推薦ウィザード」ボタンから3問の質問（業種/規模/AI活用レベル）に答えると、最適なプリセット候補がスコアリングされて提案される。

### 3-1. Phase 1: プロジェクト定義（5問）

最初のフェーズ。プロジェクトの「何を作るか」を定義する。各質問には **?アイコン**（ヘルプボタン）があり、クリックすると専門家9名のヒント（PM/デザイナー/エンジニア/マーケター等）がポップアップ表示される。

> **✨ Lv.2+の新機能**: Phase 1のステップ0に「📋 ヒアリング結果から入力」バナーが表示される。
> - **ヒアリングシートDL**: 比較ボタン隣の「📥 ヒアリングシート」ボタンでMarkdown形式のヒアリングシートをDL。クライアントに渡して情報を収集できる。
> - **ヒアリングインポート**: 入力済みMarkdownをインポートすると26問の回答が一括設定される（`showHearingImport()`）。受託開発のヒアリング結果を直接DevForgeに反映。

#### 質問①: プロジェクトの目的（purpose）
チップを選択するか自由入力。入力内容が変わるたびに**ドメイン自動推定**が走る。

```
例: 「地元農家と消費者をつなぐマーケットプレイス」
例: 「社内の技術ノウハウを蓄積・検索できるナレッジベース」
例: 「フリーランス向けの請求書作成・管理SaaS」
```
> **ポイント**: 「誰が何をできるようになるか」を一文で書く。この内容からDevForgeが**32ドメイン**（ec/saas/fintech/educationなど）を自動推定し、以降の質問・KPI・生成内容がカスタマイズされる。ドメイン自動推定は`detectDomain()`関数による32パターンマッチングで実現。purpose送信後に**ドメインヒント**（インライン `compat-info` スタイル）が表示され、推定されたドメインと関連情報が確認できる。

#### 質問②: ターゲットユーザー（target）
複数選択可。「学生」「フリーランス」「中小企業」「エンジニア」など。具体的なペルソナをイメージして選ぶ。

#### 質問③: 成功指標・KPI（success）
**動的KPIチップ自動提案**: 質問①で入力した目的から推定されたドメインに基づき、30以上のKPIプールから適切なチップが提案される。ドメインが変わるとチップも動的に更新される。スキルレベルに応じて表示数が変わる（Lv.1=6個、Lv.2=8個、Lv.3+=10個）。

```
ECなら: 💰 GMV月100万円 / 💰 CV率3%+ / 🏪 カート放棄30%以下
SaaSなら: 💰 月売上10万円(MRR) / 🔄 月間チャーン5%以下 / 🔄 DAU/MAU比率30%+
```

#### 質問④: スコープ外（scope_out）
「やらないこと」を明確に定義する。これが最も重要な質問のひとつ。

```
悪い例: 何も選ばない（スコープが無限に広がる）
良い例: 「チャット機能」「動画配信」「ネイティブアプリ」「多言語対応」を外す
```
> **なぜ重要か**: AIにこれを伝えると「それは今回作りません」という回答が得られる。スコープ外を定義しないプロジェクトは必ず機能追加地獄に陥る。

#### 質問⑤: リリース目標（deadline）
2週間/1ヶ月/3ヶ月/6ヶ月から選ぶ。生成されるWBS・タスク一覧のスプリント数と時間見積もりに影響する。

### 3-4. 互換性チェック & ガードレール

各質問に回答するたびに、**320の互換性ルール（33 ERROR + 143 WARN + 144 INFO）**がリアルタイムで評価される。ウィザード画面の下部にインラインで問題が表示される。**2,665プリセット組み合わせ**で検証済み（0 ERROR / 0 WARN）。

#### ルールの3段階

| 種別 | 色 | 意味 | 対処 |
|-----|---|------|------|
| **ERROR** | 🔴 赤 | 技術的に動作しない組み合わせ | 修正必須（Lv.2+で生成ブロック） |
| **WARN** | 🟡 黄 | 本番環境で問題になりやすい | 承知ボタンで続行可能 |
| **INFO** | 🔵 青 | 参考情報・改善提案 | 任意 |

#### ガードレール機能

- **ERRORブロック**: Lv.2以上では、未解決のERRORがある場合に「🚀 生成」ボタンが無効化される。問題を修正してから生成できる。
- **Why-Card（なぜ？カード）**: 各ルールに「▶ なぜ？」ボタンがあり、クリックすると技術的な詳細説明が展開される（例: 「PrismaはNode.js専用のコードを生成するため、Cloudflare WorkersのV8サンドボックスでは動作しません」）。
- **自動修正ボタン**: 一部のルールには「🔧 自動修正」ボタンがあり、クリックすると関連する回答を適切な値に自動変更する。
- **連鎖修正（_applyCascadingFix）**: 一つのフィールドを変更すると、依存する他のフィールドも自動的に整合性が取れるよう連鎖更新される。

#### 320ルールのカバー領域（主要カテゴリ）

| カテゴリ | ルール例 |
|---------|---------|
| 技術互換性 | Prisma+CF Workers不可、Django+Netlify不可 |
| セキュリティ設計 | MFA未設定・バックアップなし・シークレット管理なし |
| API設計 | レート制限なし・廃止計画なし・フィールド選択なし・APIバージョニングなし |
| DBスケーリング | 接続プールなし・読み取りレプリカなし・大規模向けパーティション戦略なし |
| Observability | ログ保持設定なし・アラート設定なし・ヘルスエンドポイントなし・ドメインメトリクスなし |
| 並行性 | 冪等性なし（決済）・サーキットブレーカーなし |
| FEアーキテクチャ | SPAでCSP未設定・コード分割なし・a11yツールなし |
| ドメイン固有 | 医療ドメインで暗号化なし・監査ログなし・高リスクドメインで不変条件テストなし |
| AIセーフティ | AIコード未監視・プロンプトバージョン管理なし・公平性パイプラインなし・ドリフト監視なし |
| コスト | 大規模でCDNなし・インデックスなし・ランブック未設定 |

#### compat レポート出力

互換性レポートはMarkdownにエクスポートしてClipboardにコピーするか、ブラウザの印刷機能でPDF化できる。レポートには問題点・原因・対策・検証手順が含まれる。

生成後のダッシュボードには**7項目の品質スコアカード**が表示される（✅/⬜+パーセント）:
- compat警告ゼロか / 認証設定適切か / DB設計スコア / API設計スコア / テスト設定 / セキュリティ / デプロイ設定

### ❌ 失敗1: 生成ボタンが押せない / 生成が始まらない

**原因A**: 必須項目（フロントエンド・バックエンド・データベース）が未回答
**原因B**: Lv.2以上で互換性ERRORが残っている（ガードレール機能）

```
チェックリスト:
□ Phase 2: フロントエンドを選択した
□ Phase 2: バックエンドを選択した
□ Phase 2: データベースを選択した（バックエンドが「なし」でなければ）
□ プロジェクト名を入力した（画面上部の入力欄）
□ ウィザード下部のエラー表示が0件になっている（Lv.2+の場合）
```

---

### ❌ 失敗2: ブラウザを閉じたらデータが消えた

**原因**: LocalStorageには容量制限（4MB）がある。また、ブラウザの設定によってはキャッシュクリア時に消える。

```
対策（必ずやること）:
✅ 生成直後に 📦 ZIP ダウンロード
✅ 生成直後に 📤 JSON エクスポート
✅ プロジェクトをgitリポジトリに保存

自動バックアップ確認:
→ Ctrl+K → 「IndexedDBバックアップから復元」で自動保存分を確認
```

---

### ❌ 失敗3: AIがプロジェクト内容を理解してくれない

**原因**: AIに渡すファイルの選択が間違っている、または順番が悪い

```
正しい投入順序:
1. まず .spec/constitution.md（プロジェクト憲法）を読ませる
2. 次に CLAUDE.md または AI_BRIEF.md（約1200トークン、軽量版）
3. 必要に応じて docs/01_architecture.md（アーキテクチャ概要）
4. 作業内容に応じた個別ファイル（docs/07_api_design.md など）

❌ 悪い例: ZIPを全部貼り付ける（コンテキスト超過でAIが混乱）
✅ 良い例: AIランチャーの「🤖 AI向けMD」でTOC付きに整形して渡す
```

---

### ❌ 失敗4: 「技術スタックの互換性エラー」が出る

**原因**: 相性の悪い技術の組み合わせを選択した（320のcompatルールが検出：33 ERROR + 143 WARN + 144 INFO）

```
よくある組み合わせエラー例:
❌ Flutter + NextAuth/Auth.js → Flutterはダート（Dart）製、NextAuthはNode.js専用
❌ Django + Netlify → フルサーバーはNetlifyの制限に引っかかる
❌ Remix/SvelteKit + Firebase Hosting → SSR不可（静的CDNのみ）
❌ Prisma + Cloudflare Workers → Prismaが生成するNode.js専用コードがV8サンドボックス非対応

対処法:
1. 警告メッセージに「🔧 自動修正」ボタンがある場合はクリック
2. 「▶ なぜ？」をクリックすると技術的な詳細説明を確認できる
3. なければ警告を読んで該当の選択を変更
4. ERRORルール（赤）は修正必須、WARNルール（黄）は承知して続行も可能
```

---

### ❌ 失敗5: 生成ファイルの内容がプロジェクトと合わない

**原因**: プロジェクト目的（purpose）の入力が曖昧すぎる

```
❌ 悪い例: 「便利なアプリを作りたい」
✅ 良い例: 「農家と消費者をつなぐ産直マーケットプレイス。
            農家は野菜を出品し、消費者が直接購入できる。
            Stripeで決済、Supabaseでリアルタイム在庫管理」

目的の記述が詳しいほど:
- ドメイン推定の精度が上がる（marketplace/fintech/ec等）
- KPIの提案内容が最適化される
- エンティティ定義が自動補完される
- セキュリティ要件が業種別に調整される
```

---

### ❌ 失敗6: プロジェクトが「機能追加地獄」に陥る

**原因**: スコープ外（scope_out）を定義しなかった

```
症状:
- 「チャット機能も欲しい」「多言語対応も」「モバイルアプリも」...
- AIが毎回違うアーキテクチャを提案してくる
- どこまで作ればリリースできるのか分からない

対処法:
1. DevForgeでscope_outに「チャット機能」「多言語対応」「ネイティブアプリ」を追加
2. .spec/constitution.md の「スコープ外」セクションをAIに読ませる
3. 「それは今回のスコープ外です。Phase 2で検討します」とAIに伝える根拠を持つ
```

### ❌ 失敗7: セキュリティ対策が不十分なまま本番公開

**原因**: 生成されたセキュリティドキュメントを無視した

```
見落とされがちなファイル:
- docs/43_security_intelligence.md → OWASP 2025対応チェックリスト
- docs/44_threat_model.md → 脅威モデル（データフロー図付き）
- docs/45_compliance_matrix.md → GDPR/個人情報保護法対応
- docs/46_ai_security.md → AI特有のリスク（プロンプトインジェクションなど）

本番公開前の必須チェック:
□ RLSポリシーをすべてのテーブルに適用したか
□ API認証（Bearer Token / API Key）を実装したか
□ 入力バリデーションをサーバーサイドで実装したか
□ 環境変数(.env)にシークレットを移動したか
```

---

### ❌ 失敗8: 「コードが動かない」でAIに丸投げする

**原因**: デバッグの手順が体系化されていない

```
正しい手順（AIランチャー「🔧 デバッグ支援」テンプレートを活用）:

1. エラーメッセージを error_logs.md に記録
2. DevForgeのAIランチャー「🔧 デバッグ支援」を開く
3. 生成されたプロンプト（error_logs.md参照済み）をAIに投入
4. 「5 Whys分析」で根本原因を特定
5. 修正コードを実装し、エラーログにクローズ記録
```

### ❌ 失敗9: 開発者向け — テストが壊れる

**原因**: ファイルを追加・変更したが関連するテストを更新しなかった

```
変更内容                    → 更新すべきテストファイル
─────────────────────────────────────────────────────
新ピラー追加                → test/build.test.js（関数存在確認）
                              test/gen-coherence.test.js（生成テスト）
新プリセット追加（標準）    → test/presets.test.js（件数更新）
新プリセット追加（フィールド）→ test/field-presets.test.js（件数更新）
新ドメイン追加              → test/data-coverage.test.js
互換性ルール追加            → test/compat.test.js（ルール件数更新）
ランチャーテンプレート追加  → test/skill-level.test.js（templateOrder.length更新）
KB（キーボードショートカット）追加 → src/index.html + src/core/i18n.js（同期必須）
```

### ❌ 失敗10: 開発者向け — ビルドサイズが6500KBを超える

**原因**: 大量のデータ定数を追加した

```
対処法:
1. node build.js --report でどのモジュールが大きいか確認
2. 圧縮定数の活用（_U/_SA/_T/_D など common.js の圧縮パターン参照）
3. docs/CLAUDE-REFERENCE.md の「Compression Patterns」セクションを読む
4. **why_ja/why_en の文字列短縮**: 350B/270B上限で文境界切り捨て（日本語1文字=3バイト、最大節約ポイント）
5. 現状 ~5716KB / 上限 6500KB（余裕: ~784KB）

警告ライン:
~6200KB: 警告表示
~6500KB: process.exit(1)でビルド失敗（CI hard limit）
```

## ⑦最適解トレードオフ — 設計思想と限界

### Phase 0 特化の設計思想

DevForgeは意図的に**「コードを生成しない」**ツールとして設計されている。これは制限ではなく、明確な設計判断である。

```
なぜコードを生成しないか:
1. コード生成は技術スタック・フレームワークバージョンに強く依存する
   → 数ヶ月で陳腐化する。設計書は何年も有効。

2. AIコード生成ツール（Cursor/bolt.new）が専門特化している
   → 競合より劣るコードを生成するより、設計書で補完関係を築く。

3. 「設計 → 実装」の分離がプロジェクト品質を保証する
   → 設計なきコード生成は技術的負債の温床。
```

---

### 網羅性 vs カスタマイズ性のトレードオフ

| 側面 | DevForgeの選択 | トレードオフ |
|-----|--------------|------------|
| プリセット数（860種） | 広く対応 | 特定のニッチ産業には最適化が不十分な場合あり |
| 28カテゴリ一括生成 | 包括的 | 不要なファイルも含まれる（削除して使う） |
| 32ドメイン自動推定 | 利便性 | 複合ドメイン（例: フィンテック×ヘルスケア）の推定は限界がある |
| 固定の質問フロー | 一貫性 | プロジェクト固有の独自質問は追加できない |

---

### 自動推定 vs 手動指定のバランス（32ドメイン自動推定の限界）

`detectDomain(purpose)`は32パターンのキーワードマッチングで動作する。正確な推定のために、purposeには**ドメイン特定キーワード**を含めることを推奨する。

```
✅ 高精度: 「フィンテック向け決済管理SaaS」→ fintech
✅ 高精度: 「病院の予約・電子カルテシステム」→ health
⚠️ 低精度: 「便利なビジネスアプリ」→ saas（汎用判定）
⚠️ 限界: 「農業×IoT×AI融合プラットフォーム」→ agri / iot どちら？
```

複合ドメインの場合、より影響の大きいドメインキーワードを先に書くと精度が上がる。

---

### プリセット利便性 vs 自由度

| アプローチ | 利点 | 欠点 |
|----------|------|------|
| **プリセット使用** | ワンクリックで26問自動設定、compat検証済み | 自分のユースケースに100%一致しない可能性 |
| **カスタム（白紙）** | 完全な自由 | 時間がかかる、compat違反を手動で確認 |
| **プリセット+手動調整** | **推奨**: 速度と精度のバランス | 微調整が必要 |

---

### ファイルサイズ制約（6500KB単一HTML）の影響

単一HTMLファイル設計は「インストール不要・オフライン動作」を実現するが、6500KB のサイズ制約がある（v9.15で5000KB→6500KBに引き上げ）。

```
現在: ~5716KB（87.9% of 6500KB）
残余: ~784KB

制約の影響:
- TechDB（478エントリ）はさらなる大幅拡張は慎重に
- 新機能追加にはwhy_ja/why_en短縮など既存データ最適化が必要
- データ定数は圧縮パターン（_mp()/_fpd()等）を使う必要がある
- 日本語1文字=3バイト → why_ja（≤350B・≤115文字）で節約
```

---

### スキルレベルによるUI/生成内容のトレードオフ

| スキルLv | 利点 | 欠点 |
|---------|------|------|
| Lv.0〜1（初心者モード） | 質問数最小、迷わない | 選択肢が制限される、高度な設定不可 |
| Lv.4〜6（上級者モード） | 全選択肢利用可、詳細設定 | 知識なしで選ぶと危険な組み合わせになる可能性 |

→ **推奨**: 実際の技術レベルに正直に設定する。後から上げることも可能。

### オフライン設計 vs リアルタイムAI連携

```
DevForgeの設計方針: オフライン優先
→ APIキー不要、プライバシー保護、エアギャップ対応

トレードオフ:

✅ 利点: 毎回同じ品質の出力（AI応答のブレなし）
✅ 利点: コスト0
❌ 欠点: 「ChatGPTにプロジェクト情報を学習させる」ような動的最適化はできない
❌ 欠点: 最新の技術トレンド反映は手動アップデートが必要
```

## ⑧総括的まとめ

### 生成物の効果: 225+ファイルが提供する価値の全体像

DevForgeが生成する225+ファイルは、4つの役割を担っている：

```
① AIコンテキスト層（AI Context）
   CLAUDE.md / .cursorrules / AI_BRIEF.md / AGENTS.md / skills/(7スキル)
   → すべてのAIツールがプロジェクトを即座に理解できる

② 仕様・設計層（Specification）
   .spec/ / docs/01-134_*.md / api/openapi.yaml
   → 人間とAIが共通の「プロジェクト言語」を持てる
   → DOMAIN_INVARIANTS / invariants.test.ts でドメイン整合性を保証

③ 品質保証層（Quality Assurance）
   docs/test_matrix / security_intelligence / uat_acceptance
   → リリース基準が最初から定義されている
   → 16ドメインの不変条件テスト自動生成

④ 運用準備層（Operational Readiness）
   .github/workflows/ci.yml / .devcontainer/ / scaffolding/SETUP.md
   → 「動くコードを本番環境に届ける」ための設定が完備されている
   → SOREサイクル + エラーバジェットゲート
```

### 最適な使用方法

**Phase 0 → Phase 1 フロー（推奨）**:

```
[DevForge] Phase 0: 設計・仕様・設定
    │
    ├─ 30秒で225+ファイル生成
    ├─ ZIPをプロジェクトフォルダに展開
    └─ AIツールへ投入

[Cursor/Claude Code] Phase 1: 実装
    │
    ├─ CLAUDE.md/.cursorrules を自動読み込み
    ├─ tasks.md のP0タスクから実装開始
    └─ spec/constitution.md を参照しながら開発

[bolt.new/Lovable] Phase 1 (no-code alternative):
    │
    └─ AI_BRIEF.md を貼り付けてプロジェクト説明完了
        → スキャフォールドから自動コード生成

[デプロイ] Phase 2:
    └─ .github/workflows/ci.yml でCI/CD自動化
```

**AIツール連携のベストプラクティス**:
1. 最初は `AI_BRIEF.md`（軽量版、約1200トークン）から渡す
2. 詳細が必要なら `docs/01_architecture.md` を追加
3. 実装タスクには `tasks.md` + 該当 `docs/` ファイルのみ渡す
4. フルコンテキストが必要な場合のみ「AI向けMD」を使う（トークン消費大）

### 注意点

| 制約 | 対処法 |
|-----|-------|
| **LocalStorage制限（4MB）** | 定期的にJSON/ZIPエクスポートして保存 |
| **ブラウザ依存** | Chrome/Firefox/Edge最新版を使用 |
| **生成内容の精度** | purposeに具体的なキーワードを含める |
| **多言語生成の限界** | 英語または日本語のみ（他言語は非対応） |
| **IndexedDB自動バックアップ** | Ctrl+K → 「バックアップから復元」で確認 |

### DevForgeでは不可能なこと

これを理解することで、適切な補完ツールを選べるようになる：

```
❌ コード生成（Phase 1は別ツール）
   → Cursor / Claude Code / bolt.new / Lovable / GitHub Copilot を使う

❌ リアルタイムAI連携（オフラインHTML）
   → DevForgeはオフライン完結。リアルタイムAI分析は外部ツールへ

❌ 既存プロジェクトの解析（新規プロジェクト前提）
   → 既存コードのリバースエンジニアリングは別途必要

❌ チームリアルタイム共有（ローカルファイル）
   → JSONエクスポートを共有 or Git管理で協働可能

❌ API・DB直接接続
   → 設計書を生成するのみ。実際のDB操作は実装ツールで

❌ 自動デプロイ・実行

```
